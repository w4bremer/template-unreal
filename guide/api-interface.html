# Interface

The interface is at the core of ObjectAPI (opens new window) and offers an Unreal specific implementation for handling data in projects.

Every interface is designed to be used in blueprints and C++, as we well as to provide the business logic either in blueprints or in C++. For more information on using and implementing interfaces in Blueprints check the documentation (opens new window).

In the example we have a cube which is also able to count. Therefore we call the interface CounterCube. Aside from the count property, the cube also has a color and a size. The operations increment and decrement could be used to change the count property by a certain step size. randomize could set all properties to a random value and reset restore the default values. The signal orientation could contain the orientation of the cube in the 3D space.


 
 



 












 




interfaces:
  - name: CounterCube
    properties:
      - { name: color, type: Color }
      - { name: size, type: Size }
      - { name: count, type: int }
    operations:
      - name: increment
        params:
          - { name: step, type: int }
        return:
          type: int
      - name: decrement
        params:
          - { name: step, type: int }
        return:
          type: int
      - name: randomize
      - name: reset
    signals:
      - name: orientation
        params:
          - { name: orientation, type: Orientation }

# Operations

In our example we have an increment operation specified as:


 

 

    operations:
      - name: increment
        params:
          - { name: step, type: int }
1
2
3
4

Before we can explain the declaration of the operation itself, we need to explain a few UFUNCTION (documentation) (opens new window) macro keywords.

  • BlueprintNativeEvent this function can be implemented in Unreal Blueprints
  • BlueprintCallable this function can be called from Unreal Blueprints
  • Latent this meta keyword enables functions to be run asynchronously

Every operation in the API has at least two declarations in the generated interface.

  1. The operation itself e.g. Increment(int32 Step). Due to the BlueprintNativeEvent this function is implemented by the Unreal Engine and can not be overridden in C++ by the user. This should only be overridden when inheriting in blueprints.
  2. The operation with the _Implementation postfix, e.g. Increment_Implementation(int32 Step) is a pure virtual function and has to be overridden in C++ by the user when inheriting.

In case, the operation has a return value we generate additional declarations with an Async postfix. For instance, the increment function has four different functions declared.

The complete declaration looks like this:

UFUNCTION(BlueprintNativeEvent, BlueprintCallable, Category = "ApiGear|HelloWorld|CounterCube", meta = (Latent, LatentInfo = "LatentInfo", HidePin = "WorldContextObject", DefaultToSelf = "WorldContextObject"))
void IncrementAsync(UObject* WorldContextObject, FLatentActionInfo LatentInfo, int32& Result, int32 Step);
virtual void IncrementAsync_Implementation(UObject* WorldContextObject, FLatentActionInfo LatentInfo, int32& Result, int32 Step) = 0;

UFUNCTION(BlueprintNativeEvent, BlueprintCallable, Category = "ApiGear|HelloWorld|CounterCube")
int32 Increment(int32 Step);
virtual int32 Increment_Implementation(int32 Step) = 0;

# Signals

In our example we have an Orientation signal specified as:


 

 

    signals:
      - name: orientation
        params:
          - { name: orientation, type: Orientation }
1
2
3
4

For signal handling we use dynamic multicast delegates (opens new window). These delegates must be declared before use, for instance:

DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FHelloWorldCounterCubeOrientationDelegate, const FHelloWorldOrientation&, Orientation);
1

This signal has one parameter which must be added as postfix _OneParam.

  • FHelloWorldCounterCubeOrientationDelegate is the name of the delegate type. It is constructed based on the module, interface name and the actual name of the signal from line 2 above
  • const FHelloWorldOrientation& is the type of the signal parameter and Orientation the name

WARNING

Delegates cannot be exposed to Blueprints via C++ interfaces. Therefore we must add some helper functions

All implementations of the interface can now specify this delegate and consumers can bind to it.

However, this is the pure interface declaration and we cannot use any actual type. Therefore we added getters for the delegate

UFUNCTION(Category = "ApiGear|HelloWorld|CounterCube")
virtual FHelloWorldCounterCubeOrientationDelegate& GetOrientationSignalDelegate() = 0;
1
2

Every actual implementation must return its own delegate in this function for consumers to bind to the delegate.
